# Case 3 — Gonka Blog: changelog по коммитам

> Этот кейс — часть [тестового задания](../README.md). Общий отчёт пиши в `../REPORT.md`.

## Контекст

`blog.gonkahub.com` — авто-блог про сеть Gonka (децентрализованная сеть). Часть постов пишет ИИ, часть — люди. По мере того как репо развивается, нужно вести **changelog для пользователей блога** (читатели, не разработчики). Changelog публикуется на отдельной странице `/changelog/`.

В `data/commits.json` лежат **3 свежих коммита** из публичного репо `sergeyus11/blog`. Это реальные коммиты, можешь сходить на GitHub и посмотреть полные диффы по url.

## Задача

Напиши **секцию changelog для версии v0.7.0** в формат `output/v0.7.0.md` для публикации на странице `/changelog/`.

Обязательные требования:
1. **Аудитория — нетехнические читатели блога**, не разработчики. Понятным языком, без жаргона. Не «refactored cabinet_views.py», а «теперь черновики постов в кабинете можно полностью удалить, а не просто пометить как отклонённые».
2. **Группировка** по типам: `Новое`, `Улучшения`, `Документация` (или похожие). Сами заголовки — на твой вкус.
3. **Связные** короткие предложения, не «голые буллеты». Один пункт = один факт + 1 фраза «зачем это пользователю» (где применимо).
4. **Дата релиза** — 27.04.2026.
5. **Без линков на коммиты** в публичной части (для читателей это шум). Но **в конце файла** — `<!-- internal-trace -->` секция (HTML-комментарий), где перечислены sha → к какому пункту relate, для аудита разработчиков.
6. **Markdown**, без emoji в заголовках.

## Что в кейсе

```
data/
  commits.json       ← 3 коммита (sha, message, files, ссылки)
example_done/
  v0.6.0_example.md  ← пример changelog для другой версии — образец формата
```

## Что мы оцениваем

- **Перевод тех. жаргона** — насколько понятно для нетехнического читателя
- **Группировка** — логичная или хаотичная
- **Заметил ли разный масштаб коммитов** — feature vs docs-only vs стилевой rule
- **Точность** — не переврал ли что-то из коммитов
- **Internal-trace** — корректно ли связал sha с пунктами

## Подводные камни

- Один коммит — **только обновление файла CHANGELOG.md** уже опубликованного PR. Это дубль информации в твоём списке. Как обработать — твоё решение.
- Один коммит — **внутреннее правило стиля** (`.claude/rules/gk/style.md`), это инструмент для разработчиков и AI, **читателям блога об этом знать не нужно**. В changelog оставлять или нет — твоё решение, обоснуй.
- Коммит #29 — это большой merge-комит со многими изменениями. Не пиши «merge PR #29», расскажи что внутри.
